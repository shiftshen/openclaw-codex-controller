# Communication Mechanism & Progress Visibility

## Role boundary

- `main` is the project manager / supervisor.
- The seven OpenClaw business employees are managed by `main`.
- `codex` is an **external technical employee**, not a business supervisor.
- Business-agent problems should go to `main` first. `main` decides whether to escalate a technical-development issue to `codex`.

## Canonical chains

### Business normal path
`human -> agent -> human`

### Business blocked path
`human -> agent -> main -> agent -> human`

### Technical repair / project-development path
`human -> agent -> main -> codex -> main -> agent -> human`

## Why this repo exists

This project is the public control plane for OpenClaw Main to supervise Codex professionally, while preserving a clear separation between:

- business governance
- technical implementation
- progress sensing / heartbeat visibility
- verification / acceptance

## Progress visibility rule

`acknowledged` is not equal to `working`.

A healthy controller must distinguish at least these states:

- `acknowledged`: task/message received
- `working`: there is evidence of active execution or repo activity
- `blocked`: execution cannot continue and blocker is recorded
- `done`: acceptance gate passed and evidence exists

## Evidence sources

The controller should combine two kinds of signals:

1. **Structured worker progress**
   - repo-local `reports/progress_*.json`
   - generated by `scripts/progress_report.py`
2. **Repo activity sensing**
   - `git status`
   - recently modified files
   - task output artifacts
   - verification artifacts/logs

If structured progress is missing but repo activity exists, the controller should surface that as partial visibility instead of pretending no work is happening.

## Reporting contract

Every Codex task should eventually produce:

- `status`
- `current_action`
- `changed_files`
- `verification_run`
- `blocker`
- `eta` or `next_recommended_step`
